# UX Writer 에이전트 메모리

## 핵심: 이 프로젝트 오너의 UX 기준

**취향이 아니라 원칙이다.** 오너는 UX 원칙을 정확히 이해하고 있으며, 에이전트가 그보다 높은 수준을 보여주길 기대한다. "마음에 드는 디자인"이 아니라 "원칙에 맞는 설계"를 판단한다.

## 정보 밀도의 이중 기준 (가장 중요)

### 기준 1: 반복 조회 정보 → 컴팩트하게
- **적용 대상**: 회원 목록, 대시보드, 상태 확인 화면
- **원칙**: 한눈에 파악 가능해야 함. 한 화면에 필요한 정보가 다 보여야 함.
- **예**: HomePage의 "할 일" 섹션, 회원 목록, 승인 대기 목록
- **안티패턴**: 스크롤해야 전체 상황이 보이는 대시보드

### 기준 2: 최초 인지/숙지 정보 → 하나씩 단계적으로
- **적용 대상**: 가입 프로세스, 회칙 안내, 새 기능 온보딩
- **원칙**: 한 화면에 하나의 개념만. Progressive Disclosure 엄격 적용.
- **예**: ApplyPage의 4단계 스텝, 체크리스트 항목별 설명
- **안티패턴**: 가입 폼에 모든 필드를 한 번에 보여주기

### 판단 기준
```
이 정보를 사용자가 매번 보는가?
  → YES: 컴팩트 (스캔 가능성 최우선)
  → NO (처음 보거나 중요 결정): 단계적 노출 (이해도 최우선)
```

## 프로젝트 톤앤보이스 (기존 분석)

- 기본: "~해요" 체 (친근 + 예의)
- 시스템/규정: "~습니다" 체 (공식)
- 이모지: 섹션 아이콘으로만 (본문 남발 금지)
- 구체적 수치 명시: "기본 실력" (X) → "자유형 50m 완주" (O)

## 용어 표준

- 회원 (O) / 멤버·사용자 (X)
- 가입 신청 (O) / 등록·회원가입 (X)
- 추천인 (O) / 보증인·소개자 (X)
- 수모 (O) / 수영모 (X)
- 승인 (O) / 확인(승인맥락)·허가 (X)
- 반려 (O) / 거절·거부 (X)
