# Spec Analyst 에이전트 메모리

## 설계 원칙 (프로젝트 오너 기준)

### 정보 구조 설계 시 최우선 질문
```
이 화면의 정보를 사용자가 반복 조회하는가, 최초 인지하는가?
→ 반복 조회: 한 화면에 컴팩트하게 (스캔 최적화)
→ 최초 인지: 단계별로 나눠서 (이해도 최적화)
```

### 과잉 엔지니어링 회피
- 동작하는 최소한의 구현을 먼저
- 추상화는 실제 중복이 3회 이상일 때만
- "나중에 필요할 수도"는 설계 근거가 아님

### SSOT 엄수
- 같은 정보가 두 곳에 있으면 반드시 하나로 통합
- 소스 코드가 유일한 진실, docs는 아카이브

### 데이터 설계 패턴
- 새 필드는 `optional (?)` 기본 (기존 데이터 호환)
- mockData 변경 → DATA_VERSION 증가 필수
- localStorage 기반이므로 마이그레이션 비용 항상 고려

## 프로젝트 도메인 지식

### 회원 상태 흐름
```
pending → active ↔ inactive → withdrawn
```

### 이중 승인
```
신청 → 추천인 동의(3체크) → 관리자 승인(가입비) → active
```

### 역할별 권한
- admin: 가입 승인, 회원 관리, 설정
- member (열정 이상): 추천인 가능
- member (일반): 본인 정보만
- 비회원: 소개 페이지, 가입 신청

### 이해관계자
- 총무 임미선 (admin), 코치 한선우, 수모 최선숙
